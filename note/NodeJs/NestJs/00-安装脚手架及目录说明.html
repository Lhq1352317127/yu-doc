NestJS 脚手架安装与目录结构详解 ​

一、NestJS 脚手架（Nest CLI）安装 ​

NestJS 提供了官方命令行工具（CLI）——@nestjs/cli，用于快速创建项目、生成组件（模块、控制器、服务等），极大简化开发流程。以下是详细安装步骤：

1.1 环境要求 ​

安装 Nest CLI 前，需确保本地环境满足：

  • Node.js 版本 ≥ 16.13.0（推荐使用 LTS 版本，如 18.x 或 20.x）
  • npm（≥ 8.x）、yarn（≥ 1.22.x）或 pnpm（≥ 7.x）包管理器

检查 Node.js 版本：

bash
node -v  # 输出示例：v18.18.0（需 ≥ 16.13.0）
npm -v   # 输出示例：9.8.1（需 ≥ 8.x）
1
2

1.2 安装 Nest CLI ​

通过 npm 全局安装（推荐）：

bash
npm install -g @nestjs/cli
1

如需使用 yarn 或 pnpm：

bash
# yarn
yarn global add @nestjs/cli

# pnpm
pnpm add -g @nestjs/cli

1.3 验证安装 ​

安装完成后，通过以下命令检查版本，确认安装成功：

bash
nest --version  # 输出示例：10.2.1（版本号可能因更新变化）
1

若提示“nest 不是内部或外部命令”，需检查 Node.js 环境变量配置（确保 npm 全局安装路径已添加到系统 PATH）。

二、使用 Nest CLI 创建项目 ​

2.1 初始化项目 ​

执行以下命令创建新的 NestJS 项目：

bash
nest new 项目名称  # 替换为你的项目名，如 nest-blog
1

示例：

bash
nest new nest-demo
1

2.2 选择包管理器 ​

执行命令后，CLI 会提示选择包管理器（npm、yarn、pnpm），根据个人偏好选择即可（上下键切换，回车确认）：

bash
? Which package manager would you ❤️  to use? (Use arrow keys)
> npm
  yarn
  pnpm

选择后，CLI 会自动创建项目目录、安装依赖（过程可能需要几分钟，取决于网络速度）。

2.3 启动项目验证 ​

项目创建完成后，进入目录并启动开发服务器：

bash
cd nest-demo  # 进入项目目录

三、NestJS 项目目录结构详解 ​

项目创建完成后，目录结构如下（以 nest-demo 为例）：

nest-demo/
├── src/                     # 源代码目录（核心开发区域）
│   ├── app.controller.ts    # 根控制器（处理 HTTP 请求）
│   ├── app.controller.spec.ts  # 控制器单元测试文件
│   ├── app.module.ts        # 根模块（应用入口模块）
│   ├── app.service.ts       # 根服务（封装业务逻辑）
│   └── main.ts              # 应用入口文件（启动 Nest 应用）
├── test/                    # 测试目录（集成测试）
│   ├── app.e2e-spec.ts      # 端到端测试示例
│   └── jest-e2e.json        # 端到端测试配置
├── .gitignore               # Git 忽略文件配置（如 node_modules、dist 等）
├── nest-cli.json            # Nest CLI 配置文件
├── package.json             # 项目依赖和脚本配置
├── README.md                # 项目说明文档
├── tsconfig.build.json      # TypeScript 构建配置（继承 tsconfig.json）
└── tsconfig.json            # TypeScript 基础配置

以下是关键文件/目录的详细说明：

3.1 核心源代码目录：src/ ​

src 是开发的主要区域，所有业务代码都在这里编写。

3.1.1 main.ts：应用入口文件 ​

Nest 应用的启动入口，负责创建 Nest 应用实例、配置端口、启动服务等。

typescript
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  // 创建基于 AppModule 的 Nest 应用实例
  const app = await NestFactory.create(AppModule);
  // 监听 3000 端口（默认）
  await app.listen(3000);
}
bootstrap();

3.1.2 app.module.ts：根模块 ​

Nest 应用的“根模块”，是应用的组织核心。所有其他模块、控制器、服务都需在模块中注册才能被 Nest 识别。

typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';

@Module({
  imports: [],        // 导入其他模块（如数据库模块、第三方模块）
  controllers: [AppController],  // 注册当前模块的控制器
  providers: [AppService],       // 注册当前模块的服务（可被注入）
})
export class AppModule {}

3.1.3 app.controller.ts：根控制器 ​

控制器负责处理客户端请求（如 HTTP 请求），并返回响应。通过 @Controller() 装饰器标记，内部用 HTTP 方法装饰器（@Get()、@Post() 等）定义路由。

typescript
import { Controller, Get } from '@nestjs/common';
import { AppService } from './app.service';

@Controller()  // 基础路由（空表示根路径 /）
export class AppController {
  // 注入 AppService（依赖注入）
  constructor(private readonly appService: AppService) {}

  @Get()  // 处理 GET / 请求
  getHello(): string {
    // 调用服务的方法获取数据
    return this.appService.getHello();
  }
}

3.1.4 app.service.ts：根服务 ​

服务负责封装业务逻辑（如数据处理、数据库交互等），通过 @Injectable() 装饰器标记，可被控制器或其他服务注入使用（依赖注入）。

typescript
import { Injectable } from '@nestjs/common';

@Injectable()  // 标记为“可注入”的服务
export class AppService {
  getHello(): string {
    return 'Hello World!';  // 业务逻辑：返回字符串
  }
}

3.1.5 app.controller.spec.ts：控制器测试文件 ​

用于编写控制器的单元测试，基于 Jest 框架。示例：

typescript
import { Test, TestingModule } from '@nestjs/testing';
import { AppController } from './app.controller';
import { AppService } from './app.service';

describe('AppController', () => {
  let appController: AppController;

  beforeEach(async () => {
    const app: TestingModule = await Test.createTestingModule({
      controllers: [AppController],
      providers: [AppService],
    }).compile();

    appController = app.get<AppController>(AppController);
  });

  describe('root', () => {
    it('should return "Hello World!"', () => {
      expect(appController.getHello()).toBe('Hello World!');
    });
  });
});

3.2 测试目录：test/ ​

用于存放端到端（E2E）测试代码，验证整个应用的流程（而非单个组件）。

  • app.e2e-spec.ts：端到端测试示例，通过发送 HTTP 请求测试接口。
  • jest-e2e.json：端到端测试的 Jest 配置（如测试文件匹配规则、 reporters 等）。

3.3 配置文件 ​

3.3.1 nest-cli.json：Nest CLI 配置 ​

用于配置 Nest CLI 的行为（如生成文件的模板、构建选项等）。默认内容：

json

  "sourceRoot": "src",                 // 源代码目录

3.3.2 tsconfig.json 与 tsconfig.build.json ​

  • tsconfig.json：TypeScript 基础配置（如编译目标、模块类型、类型检查规则等）。
  • tsconfig.build.json：构建时的 TypeScript 配置（继承 tsconfig.json，并排除测试文件等无需构建的内容）。

示例 tsconfig.json 核心配置：

json
{
  "compilerOptions": {
    "module": "commonjs",    // 模块系统（Node.js 常用 commonjs）
    "declaration": true,     // 生成类型声明文件（.d.ts）
    "removeComments": true,  // 编译时移除注释
    "emitDecoratorMetadata": true,  // 启用装饰器元数据（Nest 依赖）
    "experimentalDecorators": true, // 启用装饰器（Nest 核心特性）
    "target": "ES2021",      // 编译目标 ES 版本
    "sourceMap": true,       // 生成 sourceMap（调试用）
    "outDir": "./dist",      // 编译输出目录
    "rootDir": "./src",      // 源代码根目录
    "strict": true,          // 启用严格模式（类型检查更严格）
  }
}

3.3.3 package.json：项目依赖与脚本 ​

记录项目依赖（dependencies 生产依赖、devDependencies 开发依赖）和可执行脚本。

核心脚本：

json
{
  "scripts": {
    "build": "nest build",          // 构建生产版本（输出到 dist 目录）
    "format": "prettier --write \"src/**/*.ts\" \"test/**/*.ts\"",  // 格式化代码
    "start": "nest start",          // 启动应用（非开发模式，无热重载）
    "start:dev": "nest start --watch",  // 开发模式（热重载，推荐日常开发）
    "start:debug": "nest start --debug --watch",  // 调试模式
    "start:prod": "node dist/main",  // 启动生产版本（需先执行 build）
    "lint": "eslint \"{src,apps,libs,test}/**/*.ts\" --fix",  // 代码检查与修复
    "test": "jest",                // 运行单元测试
    "test:watch": "jest --watch",  // 监听模式运行单元测试
    "test:cov": "jest --coverage", // 生成测试覆盖率报告
    "test:e2e": "jest --config ./test/jest-e2e.json"  // 运行端到端测试
  }
}

四、总结 ​

通过 Nest CLI 可快速搭建 NestJS 项目，核心步骤为：

  1. 安装 Nest CLI（npm install -g @nestjs/cli）；
  2. 创建项目（nest new 项目名）；
  3. 启动开发服务器（npm run start:dev）。

项目目录中，src/ 是开发核心，包含入口文件 main.ts、根模块 app.module.ts、控制器和服务；test/ 用于端到端测试；各类配置文件（nest-cli.json、tsconfig.json 等）用于定制项目行为。

理解目录结构是后续开发的基础——模块负责组织代码，控制器处理请求，服务封装逻辑，三者配合实现清晰的职责分离，这也是 NestJS 模块化架构的核心优势。
